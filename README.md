# Main/Sub Key Security POC

This project is a Next.js proof-of-concept that separates:

- **Main key**: shared identity/reference key.
- **Sub key**: one-time usage key.

The goal is simple: keep sharing easy while enforcing strict one-time redemption.

## What this solves

Single-key systems mix identity and usage into one code. That creates replay and operational confusion.

This POC introduces a two-level model:

- Main key can be shared publicly (long or short format).
- Sub keys are consumed exactly once.
- Unused sub keys can be revoked and reissued in batches without rotating the main key.
- Audit events capture resolve/consume/issue/revoke outcomes.

## How it works

Validation uses two checks:

1. **Key integrity check**
   - Main long keys and sub keys include HMAC-derived signature parts.
   - Main short keys are deterministically derived from the long key.
2. **Database truth (Turso/libSQL)**
   - DB is the authority for key ownership and lifecycle state.
   - Consume is guarded by atomic conditions (`consumed_at IS NULL` and `revoked_at IS NULL`).

This gives practical protection against tampering and replay, while keeping the flow easy to operate.

## Key formats

- Main long: `MK` + 18 base32 chars + 8-char signature
- Main short: 12 chars
- Sub key (current): `S` + 10 base32 chars + 4-char signature
- Sub key (legacy compatibility): 6 numeric chars

## End-to-end flow

1. Generate one main key and an initial batch of sub keys.
2. Share/scanner can submit either main or sub key.
3. Resolve endpoint detects key type and validates it.
4. If sub key is used, consume endpoint marks it consumed once.
5. Operators can issue more sub keys:
   - `append`: add new keys.
   - `replace`: revoke unused keys and issue a fresh generation.

## API endpoints

- `POST /api/keys/generate`
  - Creates a main key pair plus initial sub keys.
- `POST /api/keys/resolve`
  - Resolves key type and current status.
- `POST /api/keys/consume`
  - One-time sub key consumption.
- `GET /api/keys/main/[mainId]/subkeys`
  - Operator view of sub keys for a main key.
- `POST /api/keys/main/[mainId]/subkeys/issue`
  - Issue sub keys with `append` or `replace` strategy.

## Data model summary

- `main_keys`: long key, short key, generation.
- `sub_keys`: linked to main key, consumed/revoked state, generation.
- `audit_events`: lifecycle events for observability and incident review.

## Local setup

1. Install dependencies:

```bash
npm install
```

2. Generate local environment values:

```bash
npm run setup:local
```

This creates:

- `.env.local`
- `.keys/main_sub_signing_private.pem`
- `.keys/main_sub_signing_public.openssh`
- `.keys/main_sub_signing_public.pem`

3. Fill in Turso values in `.env.local`:

- `TURSO_DATABASE_URL`
- `TURSO_AUTH_TOKEN`

`KEY_SIGNING_SECRET` is generated by the setup script.

4. Run locally:

```bash
npm run dev
```

Open [http://localhost:3000](http://localhost:3000).

## Netlify deployment

`netlify.toml` is included with `@netlify/plugin-nextjs`.
Set the same environment variables in Netlify UI:

- `TURSO_DATABASE_URL`
- `TURSO_AUTH_TOKEN`
- `KEY_SIGNING_SECRET`

## POC boundaries

This repository is a POC. Before production use, add:

- Authentication and authorization on management endpoints.
- Edge rate limiting on resolve/consume.
- Operational dashboards and alerting for abuse patterns.

## Notes

- `.env*` and key files are git-ignored.
- Re-run `npm run setup:local` if you need a fresh local secret.
